Multiple Domains or Subdomains

yourschoolsports.com or athletics.yourschool.com

Composer makes using multiple domains an subdomains super easy. You may have a site directory set up which already uses multiple domains or subdomains, or you may have multiple domains resolving to one primary domain, or this might be something that you're thinking about incoporating into your new Composer site. This guide should be able to help you congfigure everything ahead of time prior to launch. Just click on the scenario below that best matches yours.

  • "I'd like to have 2 or more domains pointing to one site"
  • "I'd like to use multiple domains for different areas of our site"
  • "I'd like to use Sub-Domains for multiple sites"

"I'd like to have 2 or more domains pointing to one site"

Sometimes website owners like to purchase and use multiple domains to catch users and point them to their Primary URL. A great example of this would be "google.com" and "google.net". Most people know Google's search page can be accessed at google.com, but Google also owns google.us, google.net, and several others. Most of the time, this is to protect brand identity or to catch users that are used to entering URLs into their browser a certain way; imagine a user typing google.net into their browser and getting a completely different site, or even worse, a site that looks like Google, but is instead a scammy or malicious site.

Google uses their most commonly know domain, google.com, as their Primary Domain. The other domains, like google.net, are used as Alias Domains. When a user types in the Primary Domain into the browser, they're brought to the Primary Domain (no surprise there, google.com goes to google.com!). However, when a user types in google.net into the browser, they are brought to google.com. The Alias Domain resolves to the Primary Domain.

What you should know

You can have several Alias domains which point to your Primary Domain in Composer. The Finalsite Support Team should be able to set those up for you at the time you launch the site or through a Support Ticket.

If you are already using your Alias Domains, the Support Team will configure those in Composer so that they'll still be active when you decide to launch the new site. Since your DNS is already configured to point to our servers, you won't have to do anything to keep them active.

If you have new domains that you're thinking of using as Alias Domains, you'll have to configure the DNS for them to match the DNS configuration of your Primary Domain. You'll also have to let the Support Team know about the Alias Domains you plan to use so we can configure them on our end. The best way to do this is to fill out a Support Ticket so one of the team members can guide you through the process, but you can also check out the DNS Configuration page for more info.

With either of these scenarios, we're using Server-Side Redirects, so the SSL certificate will be checked against any URL. You should already have an SSL certificate for your Primary Domain, but Alias Domain typically do not have SSL certificates. This means that if a user types "https" in the Alias Domain URL instead of "http" or just the bare URL, they will be shown a security warning instead of being immediately redirected to the Primary Domain. This is a highly unusual case, but still something to be aware of. For more information on SSL Certificates, see our SSL page.

"I'd like to use multiple domains for different areas of our site"

Let's say you have your main site at www.mylittlesite.com. Say you have a really important summer program that you'd like to promote, and you'd like it to have it's own domain, like www.mysummerprogram.com. In Composer, you can manage both domains under the same administration panel. Composer allows you to post content and utilize elements and modules across multiple domains, or Secondary Domains, so you can create specialized domains to handle your specialized content.

What you should know

Any Secondary Domain needs to be set up during the Deployment stage or by anyone on the Support Team. If you're currently using a Secondary Domain with your Page Manager site, you'll likely have this set up during the Deployment phase, and because the DNS is already configured for that domain to point to our servers, it will go live at the same time as the Primary Domain and you can check with your Project Manager or submit a Support Ticket to have a Secondary Domain set up for you.

Once the Secondary Domain is set up in Composer, you can add pages and post content just like you would in the Primary Domain. If you'd like the pages to be accessible under the Secondary Domain, for instance 'www.mysummerprogram.com/page', then you can create pages under that Domain the same way you would under the Primary Domain. If you'd instead prefer to have the Secondary Domain point to an area of the site under the Primary domain, for instance having 'www.mysummerprogram.com/page' redirect users to 'www'mylittlesite.com/summer-program/page', you can use Linked Pages to accomplish this. Just create a page under the Secondary Domain with a URL that you'd like to point to a page under the Primary Domain, make it a Linked page in the Page Settings, and point it to the page you'd like the user to end up on.

With either of these scenarios, we're using Server-Side Redirects, so the SSL certificate will be checked against any URL. You should already have an SSL certificate for your Primary Domain, but Secondary Domains and all other domains typically do not have SSL certificates. This means that if a user types "https" in the Secondary Domain URL instead of "http" or just the bare URL, they will be shown a security warning instead of being immediately redirected to the Primary Domain. This might not happen typically, but it's still something to be aware of. For more information on SSL Certificates, see our SSL page.

"I'd like to use Sub-Domains for multiple sites"

The most common use-case we see for sub-domains is for Districts sites or sites that encompass several organizations. Sub-domains allow several sites to be managed under one URL and on one Composer build. For example, if a school district's Primary Domain is 'www.schooldistrict.com', and there's an elementary, middle, and high school, sub-domains allow each school to have their own site URL like 'elementary.schooldistrict.com', 'middle.schooldistrict.com', and 'high.schooldistrict.com'.

What you should know

Subdomains need to be set up during the Deployment stage or by anyone on the Support Team. They work similar to the Primary Domain, in that the pages under them will inherit their URL. The URLs are something you'll want to coordinate with Support prior to launch through a Support Ticket. We need to know which subdomain URL to associate with each domain in Composer before the redesign site is launched.

If you are setting up new subdomains on your Composer site, there's a few DNS configurations you'll need to make. Unlike the Primary or Secondary domains, the DNS settings for subdomains will only require a CNAME, not an A Record. Since the A Record for the main URL is already pointing to our IP address, subdomains only require a CNAME to point to keyword.finalsite.com (where 'keyword' is your site's Finalsite Keyword). The subdomain is treated as an alias, just like 'www', so for instance a DNS setup for middle.schooldistrict.com would just be to create a CNAME where 'middle' is the alias, and it points to schooldistrict.finalsite.com (or wherever the CNAME for 'www' points to).

Subdomains may also require what's called a Wildcard SSL Certificate, which is an SSL certificate that is applied to all subdomains under a primary domain. It's best to check with your Project Manager about this during the deployment phase to see that it's included in the project build. While it's not exactly necessary, forms with payment attached to them won't work without the SSL certificate, and any user attempting to reach the subdomains over https will receive a browser warning that the page requested is not secure. For more information on SSL Certificates, see our SSL page.
